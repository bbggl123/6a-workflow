# 6A Workflow — 把"模糊需求"变成"可审计交付物"的 Agentic 开发工作流

> 一句话：**6A 给 AI Agent 套上一层"确定性外壳"**——让它不再是"你说一句、它写一堆、对错全靠运气"，而是像一支纪律严明的工程团队那样，按 `对齐 → 设计 → 拆解 → 审批 → 执行 → 验收` 六步推进，每步都有质量门控、防幻觉校验和人工关键点。

**六阶段**：`Align（对齐）→ Architect（架构）→ Atomize（原子化）→ Approve（审批）→ Automate（执行）→ Assess（验收）`

---

## 一、6A 想解决什么问题？

直接把任务丢给 AI，最常见的三种翻车：

1. **"我以为你要…"** —— 需求没对齐就开写，方向从第一步就错了。
2. **一本正经地编造** —— 缺资料时不说"我不知道",而是虚构 API、伪造数据、假装引用文献。
3. **看着完成了，其实是刷分** —— 功能"100% 完成"但塞满 TODO 占位；测试"全过"但断言全是 `assert True`；文档齐全但和代码对不上。

6A 的思路不是"让模型更聪明",而是**在模型外面构建一套工程约束**：把一个大任务强制拆成六个有门控的阶段，每个阶段不通过就打回重做，关键节点必须由人拍板，任何不确定的结论都必须标注可信度——**宁可停下来问，也不许编。**

> **核心理念**：6A 不是一条六段式流水线，而是一张 **Grounded Graph of Loops**（有锚点的环状图）。外层是刚性的六阶段骨架（不可跳过、不可乱序），内层允许灵活的重试微循环，并用不可篡改的"外部锚点"防止 AI 陷入"自己验证自己"的自嗨闭环。

## 二、六个阶段分别做什么

| 阶段 | 名称 | 干什么 | 产物 |
|---|---|---|---|
| **1** | **Align 对齐** | 把模糊需求逼成精确规范：分析项目上下文、列歧义清单、确认边界（做什么/不做什么）、定可测试的验收标准 | `ALIGNMENT_*.md`、`CONSENSUS_*.md` |
| **2** | **Architect 架构** | 先设计后编码：整体架构图、组件职责、接口契约、数据结构、异常与安全策略 | `DESIGN_*.md` |
| **3** | **Atomize 原子化** | 把大任务拆到"AI 不可能做错"的粒度：每个子任务 ≤200 行、有输入/输出契约、依赖无环 | `TASK_*.md` |
| **4** | **Approve 审批** | **人工关键点**：向你展示完整方案清单，等你明确说"确认"才动手 | 审批签字 |
| **5** | **Automate 执行** | 严格按 TASK 逐个实现：写代码 → 写单测 → 跑验证 → 更新文档，全程留痕 | 代码 + `ACCEPTANCE_*.md` |
| **6** | **Assess 验收** | 不合格就重来：需求覆盖、测试、反指标审计（防刷分）、幻觉自查 | `FINAL_*.md`、`TODO_*.md`、`TRACE_*.md` |

**铁律**：每个阶段结束都要过"质量门控",不过则**循环回退到本阶段起点重做**,绝不带病进入下一阶段。

对应到经典 SDLC：阶段 1–4 是**规划层**，阶段 5 是**执行层**，阶段 6 是**评估层**。

## 三、四个角色（职责不可混用）

即使在单个 AI 会话里，6A 也要求用"角色切换"模拟一支四人团队，**每个角色有精确职责和红线**：

| 角色 | 负责 | 红线（绝不能做） |
|---|---|---|
| **规划 Agent** | 阶段 1–4：对齐、设计、拆解 | 不写实现代码；不替你做产品/选型的最终决策 |
| **执行 Agent** | 阶段 5：按文档编码 + 单测 | 不偏离 DESIGN/TASK；不用 TODO 占位掩盖未实现 |
| **评估 Agent** | 阶段 6：验收、质量评估 | 不放宽验收标准；发现幻觉必须打回 |
| **守门员 Agent** | 跨阶段常驻：防幻觉 + 可信度校验 | 任一阶段可信度标注缺失/失真，直接判该阶段不合格 |

> 为什么要分离？因为**写代码的人和验收的人是同一个,就等于让考生自己批卷**。角色分离是防止"自我确认"的第一道结构性防线。

## 四、防幻觉：可信度标注机制（6A 的灵魂）

这是 6A 区别于普通"AI 工作流模板"的关键。任何关键假设、设计决策、外部依赖、技术选型建议，AI **必须**附上四行可信度说明：

```
【回答可信度】XX%
【内容来源】训练知识库 / 检索资料 / 项目代码 / 逻辑推测
【信息盲区】XX 内容无数据支撑
【使用建议】重要决策交叉核对
```

**并按可信度分级处置**：

| 可信度 | 处理方式 |
|---|---|
| **≥ 80%** | 正常输出，提醒交叉核对 |
| **60% – 80%** | 输出但强制标注"需人工确认",触发二级决策权交还 |
| **< 60%** | **禁止直接输出**该结论，先补资料或转成问题清单交给你 |

配套还有：**信息缺失主动告知**（不编造）、**诱导编造拦截**（识别"你就随便编一个"这类指令并拒绝）、**分段自检**（长内容前后矛盾立即停）、**全程留痕**（每次问答/拦截/标注都有唯一编号,满足可审计可追责）。

## 五、为什么是"环",不是"流水线"？

单向流水线有四个致命缺陷，6A 用四类"环"逐一防住：

| 缺陷 | 说明 | 6A 的防护环 |
|---|---|---|
| **古德哈特定律** | 指标一旦成为目标就会被刷（如为了"测试全过"写空断言） | **反指标环**（阶段6 Anti-Goodhart 审计） |
| **测量衰退** | 没人监督监督者，验收本身也会腐化 | **审计环**（阶段6 幻觉自查 + 留痕完整性校验） |
| **结构冲突** | 速度、深度、质量互相打架时没人裁决 | **仲裁环**（阶段4 分级交还 + 阶段5 预算审查） |
| **向上盲区** | AI 埋头执行，却无法质疑"目标本身对不对" | **元环**（阶段4 人工审批） |

再加上 **外部锚点体系**（Grounding Anchors）：每个阶段至少要触碰一类不可被 AI 篡改的外部验证源——**人类判断**、**物理验证**（`exit 0`/编译通过）、**冻结规则**、**数据现实**、**同行评审**；最终验收必须同时满足 ≥3 类锚点。当所有验证都来自 AI 自测时，触发"互确认警报",强制引入外部锚点。

> 完整协议（含每阶段详细步骤、门控清单、嵌套环拓扑图、锚点规则、PR/CI 治理）见 **[`6A.md`](./6A.md)** —— 这是本工作流的唯一权威来源。

## 六、这不是某个 Agent 的专属技能

6A 的核心协议 `6A.md` **与平台无关**——它就是一份任何 LLM Agent 都能读懂并遵守的指令文档。本仓库同时提供各平台的接入方式：

| 文件 / 目录 | 用途 |
|---|---|
| **`6A.md`** | **权威协议**（唯一真相源），任何 Agent 加载它即可运行 6A |
| `SKILL.md` | WorkBuddy 用的接入壳（指向 `6A.md`） |
| `adapters/` | 其他平台的即贴片段：Claude / Cursor / 通用 system prompt |
| `project-skeleton/` | 在任意项目里落地 6A 的 ISA 目录骨架 |

### 安装方式（任选其一）

**A. WorkBuddy**
```bash
git clone https://github.com/bbggl123/6a-workflow.git
cp -r 6a-workflow ~/.workbuddy/skills/6a-workflow
```
对话中输入 `6A 帮我做 xxx` 或 `@6A ...` 即激活。

**B. Claude Code / Codex（CLAUDE.md / AGENTS.md）**
1. 把 `6A.md` 和 `project-skeleton/` 放进你的项目。
2. 将 `adapters/claude.md` 内容追加进项目根的 `CLAUDE.md`（或 `AGENTS.md`）。
3. 在 Claude 里说 `6A 帮我做 xxx` 即可。

**C. Cursor**
1. 把 `6A.md` 和 `project-skeleton/` 放进项目。
2. 将 `adapters/cursor.md` 内容写入项目根 `.cursorrules`（或 `.cursor/rules/6a-workflow.mdc`）。

**D. 任意 LLM / 通用 system prompt**
1. 把 `adapters/generic.md` 整段粘进 Agent 的 system prompt。
2. 确保 Agent 能读到 `6A.md`（放在其可访问的工作区或知识库）。
3. 触发词 `@6A` / `6A` 即启动。

> 无论哪种方式，真正的"大脑"都是 `6A.md`。适配层只负责告诉 Agent"何时读它、如何激活"。

## 七、在项目中落地（project-skeleton）

把 `project-skeleton/` 整体复制到项目根，即获得一套 6A 运行骨架：

```
your-project/
├─ Agent.md              # 路由：识别 @6A 前缀、区分阶段、选择执行角色
├─ readme.md            # 执行边界、完成证据、巡视与维护规则
├─ agents/              # 四角色描述：职责、必读文档、约束、输出
├─ best-practices/      # 各角色范例与反例
├─ knowledge/           # 共享状态（红线库 / 反模式 / 成功模式 / 纠正记录）
├─ tools/               # 各角色可用/禁用工具集与约束
├─ workflows/           # 六阶段流程、嵌套环、外部锚点
└─ docs/任务名/         # 本次任务的 ALIGNMENT/CONSENSUS/DESIGN/TASK/ACCEPTANCE/FINAL/TODO/TRACE
```

其中 `knowledge/` 是**跨任务的共享记忆**：每次任务启动前先读红线库避免重复踩坑，任务结束后把新教训沉淀回去——越用越聪明。

- **五层 AI 系统架构**（Prompt → Context → Harness → Loop → Graph）
